# 代码组织与工程化规范

在编写任何业务逻辑之前，必须将“高内聚、低耦合”的工程思维贯穿始终。拒绝毫无节制的代码堆砌，注重任务拆解与逻辑复用。

## 1. 架构分层与防腐 (No God Classes)
- **拒绝上帝类/上帝组件**：绝对不允许出现动辄上千行的 `.vue` 文件或 `.ts` 文件。如果一个 Vue 组件同时负责 UI 渲染、复杂状态计算和 IPC 通信，**必须立即重构**。
- **逻辑抽离**：
    - 纯计算逻辑、时间码转换等独立提炼为 `src/utils/` 下的纯函数。
    - IPC 通信封装为独立的 Service 类或函数，避免在 Vue 组件中直接堆满 `window.motionSlice.xxx` 的调用。
    - 复杂的 Vue 响应式业务逻辑，应当抽离为 `composables/` (Hooks)。

## 2. 页面与组件分级 (UI Hierarchy)
- **Views (页面级)**：只负责布局骨架、路由承接和全局状态的向下分发。
- **Components (组件级)**：必须根据业务域细化。例如时间轴（Timeline）不能写成一个大组件，必须拆分为 `TimelineTrack`、`TimelineCursor`、`TimelineClip` 等细粒度子组件。

## 3. 恰当运用设计模式 (Design Patterns)
在处理复杂业务时，优先思考是否可以应用成熟的设计模式：
- **策略模式 (Strategy)**：例如不同的导出格式（MP4/GIF）或不同的切片规则（按时长/按大小），应抽象为统一接口下的不同策略类，避免冗长的 `if-else`。
- **观察者/发布订阅模式 (Observer/PubSub)**：主进程与渲染进程之间的进度同步，或跨组件的底层事件总线，应保持解耦。
- **工厂模式 (Factory)**：在构建不同类型的视频处理任务（如：纯转码任务、晃动识别任务）时，统一由工厂类实例化。

## 4. 命名与注释标准 (Clean Code)
- **语义化命名**：变量和方法名必须是准确的英文单词组合，宁可长不可词不达意。例如：用 `calculateShakeThreshold` 而不是 `calcThresh`。
- **注释写 Why，不写 What**：
    - ❌ 错误注释：`// 定义一个数组存切片` (代码已经表达了 What)。
    - 🟢 正确注释：`// 使用 Map 缓存切片元数据，避免在拖拽时间轴时频繁跨进程读取导致掉帧` (解释 Why)。
- **JSDoc**：对于暴露给其他模块的核心方法（特别是 `preload.ts` 中的 API），必须写明 JSDoc，包含入参类型、返回值和可能抛出的异常。
